Rust文件与I/O（三） IO

发表于 2021-11-30 更新于 2021-12-04

Read与Write

Read和Write是trait，满足这两个trait的结构体需要分别实现read与write等一系列方法。

fn read(&mut self, buf: &mut [u8]) -> Result<usize>方法传入一个类型为&mut[u8]的缓冲区，从这个缓冲区中读取数据，并返回读取数据的长度。write用法基本类似。

基础的read与write是底层的系统读写接口，这意味着直接使用read或write是很不方便的。倘若在读取或写入时出现了阻塞，则会立即终止函数调用，故我们无法立即判断读取或写入结束时IO处于何种状态，只能借助返回的usize判断。同时，read与write需要直接访问内核态，而在用户态与内核态之间频繁切换将大幅度降低程序性能。

由于直接使用read与write的缺点，我们极少直接使用read与write，而是使用封装好的含缓冲区的读写。

Seek

首先我们需要介绍Seek与BufRead。它们是非常重要的两个traits。通过Seek，我们可以控制实现了Seek trait的Reader或Writer的读写位置。

对于内存中存放的数据，我们可以自由对其进行修改。然而，读取文件则不同。考虑到一些文件可能是十分巨大的，读取文件时并不会一次性把所有文件全部读取到内存中，这对内存的开销太大了。所以，读写文件时并不能很自由地任意修改文件的内容。

一般的输入输出遵循流式传输的概念，即stream。File也可以看作一个流。这里不过多介绍流式传输。需要说明的是，传统的流式传输是单向的，而对于拥有缓冲区的流，我们可以在缓冲区中移动光标（cursor），从而达到读写文件的目的。

所谓光标的概念类似于指针，也很类似于我们手动编辑文件时，在编辑器上一闪一闪的光标。这意味这我们可以自由控制缓冲区中的数据。Seek便是Rust用来实现这个功能的，它常常配合SeekFrom枚举类使用。

值得一提的是，在io库中也有一个Cursor类，不过它与这里的光标含义不同。

Seek Trait只保证了寻址功能。要想在寻址之后方地读取数据BufReader/BufWriter。

BufReader与BufWriter

Rust中只有BufRead的trait而没有BufWrite，这是因为在rust中，含缓冲区的write与一般的write在方法上没有什么不同，只是在性能上利用缓冲区作了优化。而BufRead则包含了许多新功能，可以配合Seek使用。

Trait BufRead需要实现两个方法：

  • fn fill_buf(&mut self) -> Result<&[u8]>：返回缓冲区中的所有数据
  • fn consume(&mut self, amt: usize) 清除缓冲区中amt字节的数据。

BufRead也提供了许多有用的方法：

  • has_data_left(&mut self) -> Result<bool>
  • read_until(&mut self, byte: u8, buf: &mut Vec<u8>) -> Result<usize>
  • read_line(&mut self, buf: &mut String) -> Result<usize>
  • fn split(self, byte: u8) -> Split<Self>
  • fn lines(self) -> Lines<Self>

大部分都是我们在其他语言中非常熟悉的常用方法。需要提一下的是split与lines这两个方法。lines提供一个迭代器，每一个迭代对象都是一行的字符串。而split即常规的split方法，返回一个分割好的Vec<u8>。

Stdin与Stdout

rust中的Stdin与Stdout被封装在io库中。使用io::stdin()与io::stdout()获取标准输入输出流，其使用方法近似于File。

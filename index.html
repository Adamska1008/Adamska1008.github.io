Read和Write

Read和Write是trait，满足这两个trait的结构体需要分别实现read与write等一系列方法。

fn read(&mut self, buf: &mut [u8]) -> Result<usize>方法传入一个类型为&mut[u8]的缓冲区，从这个缓冲区中读取数据，并返回读取数据的长度。write用法基本类似。

基础的read与write是非常底层的，倘若在读取或写入时出现了阻塞，则会立即终止函数调用，故我们无法立即判断读取或写入结束时IO处于何种状态，只能借助返回的usize判断。同时，频繁使用read与write会降低速度，因为read与write常常需要直接访问内核态。

由于直接使用read与write的缺点，我们极少直接使用read与write，而是使用封装好的BufReader或BufWriter。

Seek与BufRead

首先我们需要介绍Seek与BufRead。它们是非常重要的两个traits。通过Seek，我们可以控制实现了Seek trait的Reader或Writer的读写位置。

对于内存中存放的数据，我们可以自由对其进行修改。然而，读取文件则不同。考虑到一些文件可能是十分巨大的，读取文件时并不会一次性把所有文件全部读取到内存中，这对内存的开销太大了。所以，读写文件时并不能很自由地任意修改文件的内容。

一般的输入输出遵循流式传输的概念，即stream。File也可以看作一个流。这里不过多介绍流式传输。需要说明的是，传统的流式传输是单向的，而对于拥有缓冲区的流，我们可以在缓冲区中移动光标（cursor），从而达到读写文件的目的。

所谓光标的概念类似于指针，也很类似于我们手动编辑文件时，在编辑器上一闪一闪的光标。这意味这我们可以自由控制缓冲区中的数据。Seek便是Rust用来实现这个功能的，它常常配合SeekFrom枚举类使用。

在io库中也有一个Cursor类，不过它与这里的光标含义不同。

fs基本文件操作

fs库实现了一些基本的文件操作。文件的创建与打开被封装在File类下，而对于目录的一系列操作，文件读写以及删除则以fs下的函数为入口。常用函数如下：

  • 文件内容复制 fs::copy(from, to)
  • 创建新目录 fs::create(path)
  • 读，写，删 fs::read_*(path), fs::write(path), fs::remove_*(path)
  • 文件重命名 fs::rename(path, path)
  • 设置权限 fs::set_permissions(path, perm)

File

File为文件的简单抽象，聚合了文件的一系列基本操作，如创建文件与打开文件。一些复杂的操作需配合其他结构体来完成。

MetaData

MetaData是文件的元数据。所谓元数据，包括以下内容：

  • 文件类型：是文件还是目录
  • 是否受到链接
  • 长度
  • 权限（读写）

OsString 与 OsStr

OsString 为操作系统编码相关的字符串。不同操作系统中字符串有细微差别，如Windows偏向UTF-16编码且不作安全性检查，而Linux则以UTF-8为主。Rust中字符串编码为UTF-8，要编写具有跨平台能力的软件需要考虑平台特性。与操作系统通信时，应当使用OsString获取字符串内容而不是String。这一情况主要用于获取文件路径。

OsStr与OsString的区别相当于str与String的区别。OsStr不持有数据，仅为OsString的切片。

一般我们不会主动编写OsStr，但可能需要处理标准库返回的OsStr。对此可以使用相应方法，如OsStr中的pub fn to_str(&self) -> Option<&str>转化为String或&str进行下一步处理。在转化过程中，方法会判断该OsStr是否可以合法转化为UTF-8，不合法时返回Option::None。一切从OsStr或OsString到str或String的方法都会有此判断。

Path, Component与PathBuf

我们可以直接使用字符串保存的路径打开文件，但我们总会遇到一些需要处理路径的时候，比如说提取文件名，后缀名或目录。这个时候我们需要提前封装路径并提供一些方法来简化编程。Rust标准库中提供了Path与PathBuf封装路径。

本质上来说，Path是对OsStr的封装。pub fn as_os_str(&self) -> &OsStr方法可以安全地返回&OsStr。但是我们一般不会直接使用&OsStr，故Path也提供了到&str的转换方法pub fn to_str(&self) -> Option<&str>。要想通过&str生成path，则只需使用Path::new()方法即可。

关于Path的详细内容可以查看Path文档。Path的常用方法有以下几个：

  • pub fn components(&self) -> Components<'_>：产生一个可以逐级访问每层目录的迭代器。
  • pub fn extension(&self) -> Option<&OsStr>：获取文件后缀名
  • pub fn file_name(&self) -> Option<&OsStr>：获取文件名（包含后缀名）。
  • pub fn file_prefix(&self) -> Option<&OsStr>：获取文件前缀
  • pub fn file_stem(&self) -> Option<&OsStr>：获取文件名（不包含后缀名）
  • pub fn is_dir(&self) -> bool：判断路径指向的是否为目录。
  • pub fn is_file(&self) -> bool：判断路径指向的是否为文件。
  • pub fn is_relative(&self) -> bool：判断路径是否为相对路径。

等等，大部分是比较常见且常用的功能。

需要简单提一下的是components方法，该方法返回一个Component的迭代器。

此题为线段树的经典例题，非常考验对线段树的运用。

首先，我们必定需要在线段树中存储每个区间上的最大连续子段和，这是题目要求。然而，线段树的pushup要求我们必须要从父节点的两个子节点的数据推出父节点的数据。只记录这一个值，推得了吗？推不了。我们必须维护额外的数据。

不难想到，一个区间内的最大连续子段和可能是其左右子节点对应值取个max，但不一定，因为这个子段可能会跨过mid，而子节点只能维护mid左右的数值。考虑跨过mid的情况，我们最好增加一个记录：lmax与rmax，分别表示最大前缀和和最大后缀和。最大连续子段还可能是左孩子的最大后缀和+右孩子的最大前缀和，这便是跨过mid的那个情况。

我们怎么维护lmax和rmax呢？先考虑lmax,rmax同理即可。一个区间的lmax可能不跨过mid，那么这就等同于左孩子的lmax，但也可能跨过mid，这时便是左孩子的区间和+右孩子的lmax。换句话说，我们还需要维护一个区间和。区间和显然是可以直接维护的，这个不谈。rmax同理。总而言之，我们一共需要维护四个数据，即tmax,lmax,rmax,sum。

listen()

当socket完成与地址的绑定之后，需要调用listen()进入监听状态。只有在监听状态下，socket才能响应外部的连接请求并建立连接。

int listen (int __fd, int __n)传入一个socket与一个数字，表示请求队列的长度上限。请求队列为请求的缓冲区。当socket已经建立了一个连接后，随后的连接请求将进入请求队列等待。若请求数量超过队列上线，那么请求将失败，客户端会收到ECONNREFUSED错误。

accept()

使用accept()函数从请求队列首部获取一个请求，并连接通信。

int accept (int __fd, __SOCKADDR_ARG __addr, socklen_t *__restrict __addr_len);有三个参数，第一个参数为服务器socket，第二个参数为客户端socket地址，第三个参数为客户端socket长度指针。后两个参数用于获取客户端socket的信息。我们将空的sockaddr_in和socklen_t传进去即可接收。函数返回客户端socket的句柄。

使用了accept之后，我们就可以使用write()与read()等函数进行基本的通讯了。

connect()

connect的函数用于客户端向服务器发起连接。

socket即套接字，本文不介绍基本的理论知识，仅简单说明如何在代码中使用socket。

socket()

int socket(int __domain, int __type, int __protocol)接受三个参数，返回操作套接字的句柄。三个参数分别为：

  • __domain:即通信域(communication domain)，指定了网络层的协议族，常用的如AF_INET，IPv4网络协议。
  • __type，为socket的类型，常用的有SOCK_STREAM，SOCKET_DGRAM，前者用于tcp，后者用于udp。
  • __protocol，传输层协议，即TCP,UDP等。当指定__type之后，传入形参0即可分配默认__protocol参数。

socket函数保存在<sys/socket>中。

获得socket的句柄后，我们可以使用此句柄进行write()，read() 等操作。

当我们创建了一个套接字后，这个套接字还未与一个地址连接，我们必须为套接字分配一个地址，才能够使用套接字进行通讯。为此，我们需要bind()函数。

sockaddr

在介绍bind()之前，需要先介绍sockaddr，sockaddr有两个成员变量：sin_family以及sa_data。sin_family的含义与上文中的__domain相同，同样指网络协议族。sa_data是一个14字节的char数组，用于描述这个地址，由于sin_family不同，sa_data的内容也不同。由于C不是面向对象的语言，不能通过继承体现不同sin_family下sockaddr实现的不同，所以我们通常不直接使用sockaddr，而是使用类似于sockaddr的其他结构体，在函数调用时作强制类型转换。例如，在internet环境下，使用sock_addr_in结构体。

sock_addr_in中，sa_data被替换为三个变量，分别为sin_addr，sin_port与sin_zero。sin_zero没有用处，仅用于字节对齐。sin_port为端口号，为int类型，sin_addr为ip地址，是一个结构体，其内部为s_addr，是int类型，保存了ip地址的信息。

网络字节序

sin_port与sin_addr.s_addr均为int类型。当在网络中传输数据时，规定数据以小端序进行传输。然而，许多操作系统使用大端序保存数据。

一般来说，在操作系统层面，会在网络通信时自动改变字节序。然而，对于sockaddr来说，我们需要手动设置。使用 uint16_t htonl(uint16_t) 或 uint32_t htons(uint32_t)函数，即可完成转化。前者用于32位整型，后者用于16位整型。hton表示host to net。这两个函数保存在<arpa/inet>中。

除此之外，我们还会经常使用到int inet_addr(const char*)函数。这个函数的作用是把点分割字符串表示的ip地址转化为实际的ip地址。我们书写ip时，常常使用点分割，比如127.0.0.1，但实际上的ip地址是32位字节码。虽然编写一个转换函数并不难，但标准库提供了一个转换函数inet_addr(const char*)。同时，这个函数默认输出小端序的ip地址，所以不需要另外使用htonl。

可以看到字符串被转换成了二进制ip，且确实是以小端序保存的。

此函数同样可以用于判断ip地址是否合法。若不合法，返回-1。

bind()

int bind(int __fd, const struct sockaddr *__addr, socklen_t __len)传入三个参数，若返回值为-1，则表明bind()失败。三个参数分别为socket句柄，socket将被分配的地址以及其大小。

当使用bind()之后，即对socket分配地址，socket可以进行通信了。

Pursuit

You and your friend Ilya are participating in an individual programming contest consisting of multiple stages. A contestant can get between 000 and 100100100 points, inclusive, for each stage, independently of other contestants.

Points received by contestants in different stages are used for forming overall contest results. Suppose that k stages of the contest are completed. For each contestant, k−⌊4k⌋k−\lfloor \cfrac{4}{k} \rfloork−⌊k4​⌋ stages with the highest scores are selected, and these scores are added up. This sum is the overall result of the contestant. 题解

记两个人名为 aaa 和 bbb。aaa 超过 bbb 所需要的比赛数，考虑最理想的情况，就是 aaa 全部 100100100，bbb 全部000。

每一场新比赛后，我们需要统计两人所有比赛前 kkk 大得分和。对于 bbb 来说，没有问题，因为 bbb 接下来的得分全部都是 000，那么他前 kkk 大得分只能从前 nnn 场比赛中来，等 k>nk>nk>n 之后，统分时加 000 即可。

但对于 aaa 来说，统计前 kkk 大得分会比较烦。假如 kkk 没有增加，我们就需要先减去之前统计过的得分中最小的那个，再加上 100100100。这意味着我们需要维护的是 aaa 得分中的最小值，而不是最大值。

于是我们用优先队列维护各自的前 nnn 个数据，其中 bbb 是大根堆，而 aaa 是小根堆。对于 bbb，我们先求出前 kkk 大得分之和并令其出队，若 kkk 变化，继续从堆中取出得分并累加。对于 aaa，我们先求出其所有得分之和，然后统计前 n−kn-kn−k 小的得分，相减得到前 kkk 大得分。每一场新比赛后都先把总得分加 100100100，若 kkk 无变化，也就意味着n−kn-kn−k 变化，则从堆中取出一个最小值进行累加。

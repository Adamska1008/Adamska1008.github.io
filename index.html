虚拟化CPU

关键词：

  • 上下文切换
  • 调度策略

进程

定义：运行的程序。

进程的构成：机器状态

  • 地址空间（可访问内存）
  • 寄存器信息
    • 通用寄存器
    • 特殊寄存器：
      • 程序计数器（Program Counter, PC），用于管理指令执行
      • 栈指针(stack pointer, %rsp in x86_64 asm) 与 帧指针(frame pointer, %rbp in x86_64 asm)用于管理参数栈，局部变量，返回地址。
  • I/O信息

创建进程的基本过程

  1. 将代码与所有静态数据从存储器加载到地址空间上
  2. 操作系统自行（或手动）为程序的运行时栈（run-time stack 或 stack）分配内存。
  3. 可能为堆分配内存
  4. 其他初始化任务，例如：创建标准输入、输出、错误流

进程状态
至少包括

  • 运行
  • 就绪
  • 阻塞

或更加复杂

数据结构-进程列表（process list）
为了跟踪进程信息，需要使用一个数据结构来记录所有进程的信息。当一个进程停止时，将在这个数据结构中保存对应的机器状态，从而使得这个进程可以被恢复。这就是上下文切换。

记录一些我觉得还不错的软件（大部分软件同时开源）

通用软件

Hexo

此为Hexo的GitHub仓库。搭建博客的框架，同样也是搭建我这个博客的框架（主题为NEXT）。特点是轻量化且扩展性强。由于个人比较懒，所以本博客许多功能都未搭建完成，故先不详细介绍Hexo，此处暂且跳过。

Clash

此为Clash的GitHub仓库。此为ClashForWindows的GitHub仓库。此为ClashForAndroid的GitHub仓库。用的非常爽的科学上网软件。特色是可以不借助软路由直接开启局域网科学上网。使用GoLang编写。这是否也在侧面证明了GoLang在网络应用方面的强大能力呢？

WINDOWS

GIMP

一句话评价：简单版photoshop。众所周知，photoshop是平面设计领域无可争议的王者软件，但对于普通用户来说，无论是其高昂的价格还是复杂的使用逻辑都相当地劝退。

如果说有什么软件能代替photoshop的话，那一定是GIMP。GIMP全称为GNU Image Manipulation Program。（对，它也是GNU计划的一部分）。该软件具有相当多的优点：

  • 开源且免费，所有代码均可在GIMP的Gitlab主页上找到。
  • 支持使用python脚本进行编程。python是一种易于使用的脚本语言。大部分软件开发者都可以非常迅速的学会python的基本语法。GIMP支持通过python脚本修改图片。
  • 功能强大，根据Franklin Veaux的Quora回答以及我个人的使用体验来看，基本上来说，我想要做的图片处理都可以非常轻松快速地完成。
  • 相对而言更加轻量化，例如启动速度较快，编辑响应速度较快。并没有比较过大文件（例如GB大小级别的图片）的响应时间，这是由于对于我这个普通用户来说，并没有这方面的需求。仅以日常使用而言，GIMP是更加方便趁手的。

Grammly

一个轻量级且易于使用的语法检查软件（当然，是英文语法检查）。此为官网地址。安装好软件之后，所有在浏览器中编写的英文文本都将自动进行语法检查，且支持自动修改。软件分付费与免费两个版本。以我个人体验来看，免费版本即可满足我个人的所有需求。

Motrix

开源的下载软件，此为Motrix的Github仓库。使用了Node相关的一系列技术栈来搭建，包括Electron和Vue。整体使用感觉良好。Chrome有Motrix的Web插件，但体验一般，非常可惜。该软件优点如下：

  • 下载速度不错。水平一般能力有限，不分析原因。仅为使用感受。一般文件的下载速度较快。
  • 界面清爽简介。这点纯粹是主观感受。但跟IDM和迅雷比一下就能明白，好看的外观绝对是加分项。
  • 支持各类常见协议的下载方式，除了HTTPS, BitTorrent之外，还支持RPC，这个就相对而言比较少见了。但是对我来说很有用。至于为什么有用这点暂且按住不表。甚至兼容迅雷的thunder协议，这点懂得都懂。
  • 允许使用代理服务器进行缓存下载。虽然我并不需要（有许多更好的解决措施），但有些人可能有这方面的需求。

Captura

开源的视频录像软件，此为Captura的Github仓库。

Notion

我见过的最为简洁，轻量化而又功能强大的To-do-list应用。此为Notion官网。我个人相当建议使用Notion来规划个人安排。

很经典的一道最小生成树的题目。题目要求给定一颗树，要求扩展为完全图之后该树为唯一最小生成树。我们使用构造法来解决这道题目。

在kruskal算法中，每次合并两个点集时，只会在两个点集的最短路上建边。由于这道题中我们求的是完全图，所以我们可以考虑在合并集合时把两个集合中所有点连上边。边权为多少时符合题意呢？假设原边权为 www ，其余点之间边权为 w′w'w′，若 w′≤ww' \le ww′≤w，显然我们在合并集合时可以选择 w′w'w′ 而非 www，这使得最小生成树不是原来的树。而由于题目中要求总 w′w'w′ 尽可能小，故 w′w'w′ 取 w+1w+1w+1。这里没有证充分性，但我感觉还是比较显然的。

Read与Write

Read和Write是trait，满足这两个trait的结构体需要分别实现read与write等一系列方法。

fn read(&mut self, buf: &mut [u8]) -> Result<usize>方法传入一个类型为&mut[u8]的缓冲区，从这个缓冲区中读取数据，并返回读取数据的长度。write用法基本类似。

基础的read与write是底层的系统读写接口，这意味着直接使用read或write是很不方便的。倘若在读取或写入时出现了阻塞，则会立即终止函数调用，故我们无法立即判断读取或写入结束时IO处于何种状态，只能借助返回的usize判断。同时，read与write需要直接访问内核态，而在用户态与内核态之间频繁切换将大幅度降低程序性能。

由于直接使用read与write的缺点，我们极少直接使用read与write，而是使用封装好的含缓冲区的读写。

BufReader与BufWriter

Rust中只有BufRead的trait而没有BufWrite，这是因为在rust中，含缓冲区的write与一般的write在方法上没有什么不同，只是在性能上利用缓冲区作了优化。而BufRead则包含了许多新功能，可以配合Seek使用。BufReader实现了BufRead。

实现Trait BufRead需要实现两个方法：

  • fn fill_buf(&mut self) -> Result<&[u8]>：返回缓冲区中的所有数据
  • fn consume(&mut self, amt: usize) 清除缓冲区中amt字节的数据。

BufRead也提供了许多有用的方法：

  • has_data_left(&mut self) -> Result<bool>
  • read_until(&mut self, byte: u8, buf: &mut Vec<u8>) -> Result<usize>
  • read_line(&mut self, buf: &mut String) -> Result<usize>
  • fn lines(self) -> Lines<Self>

使用read_to_string()方法把文件中的所有内容都读取到了字符串中。

若你的文件内容需要以行分割，则可以使用lines()方法。

但是，若只使用BufReader操作，我们只能够将一个文件看作一整个缓冲区进行整体操作，如果想要精确地改变某些位置的某些值，必须将文件全部读入内存中，之后再写回去，这显然不是我们希望的。我们希望的是直接对文件进行精确地操作，例如精确地只读取文件中第20个字节到40个字节的数据。这需要配合Seek与Take使用。

Seek与Take

首先我们需要介绍Seek。通过Seek，我们可以控制实现了Seek trait的Reader或Writer的读写位置。

对于内存中存放的数据，我们可以自由对其进行修改。然而，读取文件则不同。考虑到一些文件可能是十分巨大的，一般来说，读取文件时并不会一次性把所有文件全部读取到内存中，这对内存的开销太大了。

一般的输入输出遵循流式传输的概念，即stream。在这个情况下，我们一次只操作一部分流过程序的数据，故内存中的数据不会过于大，例如lines()方法，提供了以行为单位的流。这种IO方式虽然节约了内存，但是修改文件内容却很不方便。

Seek则优化了读取流的方式。通过Seek，我们可以快速定位到流的某一个位置进行修改。它常常配合SeekFrom枚举类使用。

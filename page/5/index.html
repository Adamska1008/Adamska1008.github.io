Linux C中基于句柄的文件操作

发表于 2021-10-20 更新于 2022-02-12

fcntl.h 与 unistd.h 定义了一系列基于句柄的文件操作函数以及常量。其读写文件基于句柄，与stdio.h中基于file的文件读写有很大的不同。考虑到Linux一切皆文件的特性，fcntl.h有着更好的泛用性，且更接近底层。

虽然这些操作更加接近底层，但却未必比直接使用库函数来的快速，因为从用户态切换到内核态需要额外的时间。在实际使用过程中，最好针对此点进行优化。

句柄（文件描述符）

句柄（Handle）是一个用于标识文件的标识符，其在C中为int类型。句柄在Linux中，不仅可以标识狭义的文件，还可以标识输入输出流或套接字。

标准输入流的句柄为0，标准输出流的句柄为1，标准错误流的句柄为2，对应的常量作为STDIN_FILENO, STDOUT_FILENO, STDERR_FILENO存储在fcntl.h中

write()

ssize_t write(int fildes, const void* buf, size_t nbytes)从缓冲区buf中向句柄指向的文件写入nbytes的数据。在一般的文件中，若nbytes大于文件本身的长度，则文件将被扩容。特别地，若句柄指向pipeline或一些具有限制的媒介中，数据量超过容量的部分将被延迟接收，直到获得充足的容量位置。

write返回中断前被接受的字节数。如果write在没有写入任何字节的情况下被中断，将返回-1，并将errno设置为EINTR。特别地，若nbytes == 0，则返回0，并不进行任何操作。

write写入文件的方式由open函数的参数决定。若为O_APPEND，则在文件末尾续写数据，若为O_TRUNC，则会覆盖文件本身的数据。特别地，对于pipeline，所有写入将直接附加到pipeline的末尾。

open()

要为一个句柄赋值，可以使用open打开文件，或者socket构建套接字。新的句柄从3开始计数，不同进程有着独立的句柄计数空间。

int open(const char *path, int oflag,...)表示打开path对应的文件，参数为oflag，返回这个文件持有的句柄。oflag的具体内容可以查阅open词条。open函数建立起文件与句柄之间的联系，open返回的句柄为可用句柄中的最小值，即从3,4,5…计数。当一个句柄被关闭后，其将被销毁，新生成的句柄将覆盖之前的句柄，例如：

句柄同样可以使用socket(),accept(),connect()等函数构建，这部分的内容会放在之后的文章中。

当句柄构建失败时，返回值为-1。

read()

同样的，可以使用read()函数进行读取操作

ssize_t read(int fildes, void *buf, size_t nbyte)从句柄指向的文件中读取nbytes的数据到buf中。

close()

在文件读写结束之后，使用int close(int fildes);关闭管道，防止资源泄露，同时，close一个句柄也意味着一切对文件的锁都将被解除。close一个pipeline时，其中的所有内容将被遗弃。返回0表示关闭成功，-1表示关闭操作被中断。

流式的read(), write()

read()与write()函数是基于流的读写操作，这意味着数据的收发是无边界的。这也意味着read()与write()是适用于TCP的。以下方的程序为例

可以看到，我们分两次向标准输出流调用了write()，而只需要调用一次read()就能收到之前发送的所有消息，这便是流式传输。

实际上，write()在调用之后并非立即传送数据，而是先将数据移送至缓冲区，read()同样也是直接向缓冲区读取数据。

read()的BUF_SIZE即为可从缓冲区读取数据的上限，虽然我们设置BUF_SIZE为1024，但并不一定能读取到1024字节的数据，因为这只是读取数据的上限。

调用一次read()时，读取到缓冲区为空或读取的数据量达到上限为止。

即使pipeline两端中一方关闭pipeline，缓冲区也不会消失，而是直到两端都关闭时才消失。

模板-图论

发表于 2021-08-21 更新于 2022-02-13 分类于 算法竞赛 ， 模板

邻接表存图

拓补排序

将所有入度为0的节点入队。每次入队时，更新相邻节点的度数，并判断跟新后是否为0。

朴素Dijstra

在所有未确定最短路的点中，寻找dist最小的点，用这个点更新所有相邻节点，并确定该点距离已经为最短。

堆优化Dijstra

使用堆维护距离与相应的节点。

Bellman-Ford算法

遍历所有边n次，对于每条边{a, b, w}，判断b是否能被a与w更新。

spfa 算法

考虑到在BF算法中，只有当一个节点被更新过后才可能用其更新相邻节点，所以记录哪些节点被更新过，并且只从这些节点中取点更新相邻节点。

floyd算法

考虑到任何一条路径都可以转化为两条更短的路径，所以通过枚举拆分路径的方式维护最短路。

朴素版prim算法

从一个点开始扩张生成树。每次选取生成树的相邻节点中最近的节点放在生成树中，最后得到的必定是最小生成树。

Kruskal算法

枚举从短到长的所有边，逐个挑选。若选取了某条边后会形成环，则不取这条边，否则取。

染色法判断二分图

考虑到二分图等价于图中是否存在偶数环，故选取两种颜色，相邻节点染不同的颜色，当出现矛盾时，则不存在偶数环，故不为二分图。

匈牙利算法求二分图最大匹配

将二分图划分为左右两侧。对于每个左侧节点，逐个考虑是否可以连接右侧节点。若右侧节点不存在连接，则直接连接。若右侧节点存在连接，则检测该连接对应的左侧节点是否可以重新分配右侧节点。需要注意的是，当需要重新分配时，应直接从下一个右侧节点开始检测而非检测所有右侧节点。

二分图中求最小点覆盖、最大独立集与最大团

可以证明二分图中最小点覆盖等于最大匹配数，所以直接求一遍最大匹配即可。

在二分图中，求最大独立集只需把最小点覆盖去掉即可，故解为 n−mn - mn−m （mmm 为最小点覆盖）。否则为NP问题。

最大团与最大独立集为互补关系。

LCA (倍增 在线)

预处理 fa(i,j)fa(i, j)fa(i,j) 表示从节点 iii 开始往上跳 2j2^j2j 步后可以到达的距离。使用二进制拼凑的方法进行跳跃，先将两点跳到统一深度上，然后再跳到祖先节点之前。

LCA(tarjan 离线)

在一次dfs中，将所有节点分为三类：未遍历，遍历中，遍历结束。在遍历的过程中，若要查找某个遍历结束的节点和遍历中的节点的LCA，只需要在正在遍历的路径上找即可。使用并查集维护遍历结束的节点在遍历中的节点的路径上的父节点。

动态链接

假设一个操作系统上运行着100个程序，每个程序都需要使用printf。如果使用静态链接库，则printf的代码就会被复制100次，这显然造成了资源浪费。

共享库(shared library)解决了这一问题。它可以加载到任意内存地址，并和一个程序链接起来，这就是动态链接。在Linux中为.so文件，在Windows中为.dll文件。所有引用共享库的程序共享一个so文件中的代码和数据。

依旧使用上篇中提到的add.c和mult.c作为案例。

C语言从应用程序中加载和链接共享库

要想链接共享库，既可以编译时加入编译参数，也可以在程序执行的过程中动态地加载共享库。

Linux系统为动态链接器提供了一个简单的接口，在<dlfcn.h>库下。

发表于 2021-08-11 更新于 2022-02-12 分类于 静态链接与动态链接

本系列仅简单介绍链接的概念与基本使用，不包括链接的实现机制。

何为链接

链接是多文件编程的基础。链接器以可重定位目标文件(.o文件)与命令行参数作为输入，输出可执行目标文件。

符号：在可重定位目标文件中，一个符号对应了一个函数、全局变量或全局静态变量。局部变量没有符号对应，因为局部变量在栈上完成内存管理。

链接器主要完成两个任务：

  • 符号解析。可重定位目标文件中定义或引用了符号。链接器需要为不同文件中的引用寻找定义，并将其关联起来。
  • 重定位。编译器和汇编器生成从地址0开始的代码和数据节(section)。在链接时，需要为符号重新分配合适的内存地址。

静态链接

当使用链接时，我们难免遇到一种情况：我们只需要引用.o文件中的一个函数，却必须将整个文件链接进来。

假设一个math.o，定义了下列函数

我们只需要使用max函数，却必须一次性复制五个函数到执行文件中，这就造成了内存的浪费。

一个可行的解决办法是，一个.o文件只定义一个函数。如printf.o只定义printf，scanf.o只定义scanf。但这又会带来问题：为每一个函数添加链接是很繁琐的。一个程序可能会调用数十个函数，链接几十个函数既麻烦，也不好记忆。

因此，我们需要一种可以被链接器链接的文件。把所有目标文件存放在一个文件中。当需要引用这个函数时，只复制一个函数对应的一个目标文件，而不会复制其他函数。

这种文件就是静态库(static library)，在Windows中为.lib文件(library)，在Linux中为.a文件(archive)。

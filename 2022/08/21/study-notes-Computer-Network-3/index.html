计算机网络笔记（三）- 传输层

发表于 2022-08-21 更新于 2022-08-26 分类于 《计算机网络：自顶向下方法》学习笔记

计算机网络-传输层知识点

传输层服务的基本理论和基本机制：

  • 多路复用/分用
  • 可靠数据传输机制
  • 流量控制机制
  • 拥塞控制机制
    Internet的传输层协议
  • UDP：无连接传输服务
  • TCP：面向连接传输服务
  • TCP拥塞控制

传输层协议为运行在不同Host上的进程提供了一种逻辑通信机制。

端系统运行传输层协议：

  • 发送方：将应用递交的消息分成一个或多个Segment向下传给网络层。
  • 接收方：将接收到的Segment组装成消息，并向上交给应用层。

传输层位于网络层之上，依赖于网络层服务，对网络层服务（可能的）加强，提供应用进程之间的逻辑通信机制。

Internet传输层协议

  • 可靠、按需的交付服务（TCP）

    • 拥塞控制
    • 流量控制
    • 连接建立

  • 不可靠的交付服务（UDP）

    • 基于“尽力而为（Best-effort）”的网络层协议

两种服务都不提供延迟和带宽的保障

多路复用和多路分用

如果某层的一个协议对应直接上层的多个协议/实体，则需要复用/分用。

接收端进行多路分用：传输层依据头部信息将收到的Segment交给正确的Socket，即不同的进程。

从多个Socket接收数据，为每块数据分装上头部信息，生成Segment，交给网络层。

工作方式
主机接收到IP数据报（datagram），每个数据报携带源IP地址、目的IP地址，一个传输层的段。每个段携带源端口号和目的端口号。

主机收到Segment后，传输层协议提取IP地址和端口号信息，将Segment导向相应的Socket。

无连接分用

UDP的Socket用二元组（目的IP地址，目的端口号）标识，主机收到UDP段后检查段中的目的端口号，将UDP段导向绑定在该端口号的Socket。来自不同源IP地址和/或源端口号的IP数据包被导向同一个Socket。

面向连接的分用

TCP的Socket用四元组标识（源IP地址，源端口号，目的IP地址，目的端口号），接收端利用所有四个值将Segment导向Socket，服务器可能支持多个TCP Socket。

UDP协议

User Datagram Protocol

  • 基于Internet IP协议，实现了多路复用/分用，简单的错误校验机制（没有错误恢复）。
  • UDP段可能丢失或非按序到达。
  • 无连接，发送方接收方不需要握手，每个UDP段的处理独立于其他段。

优势：

  • 无需建立连接，减少延迟
  • 实现简单，无需维护连接状态
  • 头部开销少（8字节）
  • 没有拥塞控制，应用可以更好地控制发送时间和发送速率

常用于流媒体应用，容忍丢失，速率敏感。

报文段格式

  1. 源端口号、目的端口号（共32字节）
  2. 长度，错误校验和（共32字节）
  3. 数据

校验和（check sum）
目的：检测UDP段在传输中是否发生错误（如位翻转）

发送方

  • 将段的内容视为16-bit整数
  • 校验和计算：计算所有整数的和，进位加在和的后面，将得到的值按位求反，得到校验和
  • 放入校验和字段

接收方

  • 计算所收到的段的校验和
  • 将其与校验和字段进行对比
    • 不相等：检测出错误
    • 相等：没有检测出是错误（但可能有错误）

可靠数据传输

可靠数据传输协议（rdt）位于传输层，应用层调用rdt_send()函数将数据交付给rdt，而rdt调用udt_send()，在不可靠的信道（IP协议）上传输数据，接收方使用rdt_rcv()处理数据，最后使用deliver_data()向上层应用交付数据。其中控制信息是双向流动的。

Rdt 1.0：可靠信道上的可靠数据传输

假设底层信道完全可靠：不会发生错误、不会丢弃分组。发送方和接收方的FSM独立。

发送方只有一个状态，等待上层调用，对于rdt调用，打包后直接进行udt_send。接收方只有一个状态，等待下层的调用，对于一个数据包直接解包并向上层deliver_data。

Rdt 2.0：产生位错误的信道

底层信道可能翻转分组中的位：利用校验和检测位错误。

如何错误中恢复？

  • 确认机制（Acknowledgements,ACK）：接收方显式地告知发送方分组已正确接收
  • NAK：接收方显式地告知发送方分组有错误
  • 发送方收到NAK后，重传分组

基于这种重传机制的rdt协议为ARQ协议（Automatic Repeat reQuest） 议。引入差错检测、反馈控制消息、重传。

发送方状态机有两种状态。第一个状态等待调用，打包并发送后进入状态二，等待对方控制消息，若NAK则重传，若ACK等待上层下一次调用。

接收方状态机有一种状态，等待下层调用。若发现有错误，反馈NAK，否则反馈ACK并解包deliver_data。

缺陷：确认消息可能被破坏。

Rdt 2.1和2.2

Rdt2.1：应对ACK/NAK破坏，状态需要记住序列号，导致状态翻倍。

发送方：使用序列号0、1标记分组。当接收到ACK时，翻转序列号并发送下一个分组。否则重传。

接收方：分别等待0、1分组的数据，若分组损坏或序列号错误，损坏时发NAK；错误时发ACK，但是不上传错误序列号的数据。

Rdt2.2：无NAK消息协议。

接收方通过ACK告知最后一个被正确接收的分组。在ACK消息中显式加入序列号。

Rdt 3.0

假定信道既可能发生错误，也可能丢失分组。

方法：发送方等待合理时间

  • 如果没收到ACK，重传
  • 如果分组或ACK只是延迟而不是丢了
    • 重传会产生重复，序列号机制
    • 接收方需要在ACK中显式确认分组
  • 需要定时器

发送方状态：首先启动定时器并发包，若定时器到时，重传并重置定时器。

可能情况：

  • 数据丢包，重传
  • ACK丢包，重传，接收方依据序列号删除重复分组
  • 延迟过长，接收方发送两次ACK，依据序列号删除重复分组

Rdt 3.0能够正确工作，但性能很差。

流水线机制与滑动窗口协议

允许发送方在收到ACK之前连续发送多个分组

  • 更大的序列号范围
  • 发送方和/或接收方需要更多缓存

实现流水线机制需要滑动窗口协议
窗口：

  • 允许使用的序列号范围
  • 窗口尺寸为N：最多有N个等待确认的消息

滑动窗口

  • 随着协议的运行，窗口在序列号空间内向前滑动

GBN

Go-Back-N协议

  • 分组头部包含k-bit数据
  • 窗口尺寸为N，最多允许N个分组未确认
  • ACK(n)：确认到序列号n（包含n）的分组均已被正确接收
    • 可能收到重复ACK
    • 只需要记住唯一的expectedseqnum
  • 乱序到达的分组
    • 直接丢弃
    • 重新确认序列号最大的、按序到达的分组
  • 为空中分组设置计时器
  • Timeout(n)事件：重传序列号大于等于n，还未收到ACK的所有分组

发送方扩展FSM：只有一个状态。

SR

Selective Repeat协议

GBN会重传所有ACK(n)之后的分组，影响性能。SR协议单独确认分组，不丢弃乱序分组，设置缓存机制。

发送方只重传没有收到ACK的分组，为每个分组设置定时器

接收方同样需要维护窗口，两个窗口不同步。

发送方：

  • 要求发送数据：若窗口有空，发包
  • timeout：重传，重设定时器
  • ACK(n)：标记包已收到，移动窗口

接收方：

  • 收到包：发送ACK(n)，若乱序则缓存，否则交付数据并滑动窗口
  • 收到不在序列之间的包：仍发送ACK

缺陷：重用序列号时，不一定能够分辨相同序列号对应的包是否为同一个。要求双方窗口的尺寸之和小于序列号范围。

TCP协议

特点：

  • 点对点
  • 可靠的、按序的字节流
  • 流水线机制：拥塞控制和流量控制机制
  • 发送方/接收方缓存
  • 全双工：同一连接中能传输双向数据流
  • 面向连接：
    • 通信双方在发送数据之前必须建立连接
    • 连接状态只在连接的两端中维护，在沿途节点中并不维护状态
    • 包括缓存、连接状态变量、socket

段结构

  • 源端口号、目的端口号
  • 序列号（不是段的编号，而是段中第一个字节的编号，建立连接时随机选择）
  • ACK序列号（希望接收到的下一个字节的序列号，之前的所有字节均被正确接收）
  • 头部长度、标志位、接收窗口
  • 校验和

可靠数据传输

  • 流水线机制
  • 累计确认
  • 单一重传计时器

RTT与超时：定时器重传时间大于RTT，使用SampleRTT测量RTT，求多个取平均值。ERTT=(1−α)E_RTT+αS_RTTE_RTT = (1-\alpha)E\_RTT+\alpha S\_RTTER​TT=(1−α)E_RTT+αS_RTT。

超时时间的设置：E_RTT+安全边界。变化大则设置较大的边界。变化由两RTT的差值决定

TCP发送方事件
收到数据：

  • 创建段
  • 开启计时器
    超时
  • 重传
  • 重设计时器
    收到ACK 如果未确认
  • 更新SendBase
  • 重设定时器

流量控制

接收方为TCP连接分配buffer，发送方传输太快可能导致buffer溢出。流量控制是一种速度匹配机制。

接收方通过在段中的头部字段将RcvWindow告诉Sender，发送方限制自己已经发送但还未接收的数据不超过接收方RcvWindow尺寸。

连接管理

传输前需要建立连接。

  1. 客户机发送SYN段
    • 无数据
    • 选择初始序列号
  2. 服务器收到SYN段，答复SYNACK段
    • 分配缓存
    • 选择初始序列号
  3. 发送ACK段，可能发送数据

关闭连接：

  1. 客户机发送FIN段
  2. 服务器收到FIN，回复ACK，并发送FIN
  3. 客户端收到FIN，回复ACK
  4. 服务器收到ACK，连接关闭

拥塞控制原理

非正式定义：太多主机发送了太多数据或发送速度太快，以至于网络无法处理

表现

  • 分组丢失（路由器缓存溢出，重传导致吞吐率变低）
  • 分组延迟过大（在路由器缓存中排队）

拥塞控制的方法：

  • 端到端的拥塞控制
    • 网络层不需要显式的提供支持
    • 端系统通过观察loss，delay等网络行为判断是否发生拥塞
    • TCP采用这种方法
  • 网络辅助的拥塞控制
    • 路由器向发送方显式地反馈网络拥塞信息
    • 简单的拥塞指示

案例：ATM ABR 拥塞控制
如果发送方不拥塞，使用可用带宽；如果发送方路径拥塞，将发送速率降低到最低速率。交换机设置RM（资源管理） cell位，NI bit：rate不许增长；CI bit：拥塞指示。RM cell由接收方返回给发送方。

TCP拥塞控制

发送方限定发送速率，假定拥塞窗口大小CongWin，LastSent-LastAcked <= CongWin。rate = CongWin/RTT。

定义Loss事件：超时或3个重复ACK，发生之后发送方降低速率。

使用两个机制调整速率

  • 加性增-乘性减：谨慎探测可用带宽，直到发生loss
    • 方法：每个RTT将CongWin增大一个MSS（最大段长度）。发生loss后将CongWin减半。
  • 慢启动：可用带宽可能远远高于初始速率，当连接开始时指数性增长。
    • Threshold变量：当CongWin达到Loss事件前值的一半。达到Threshold从指数性增长切换为线性增长。

Loss 事件处理：
3个重复ACKs：
* CongWin切到一半
* 线性增长
Timeout：
* CongWin设置为1个MSS
* 开始时指数增长，设置Threshold值

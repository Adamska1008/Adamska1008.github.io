计算机网络笔记（二）-应用层

发表于 2022-07-26 更新于 2023-01-14 分类于 《计算机网络：自顶向下方法》学习笔记

计算机网络-应用层知识点

网络应用的体系结构：

  1. 客户机/服务器结构（Client-Server，C/S）
  2. 点对点结构（Peer-to-peer，P2P）
  3. 混合结构（Hybrid）

客户机/服务器结构

服务器特点

  1. 7*24小时提供服务
  2. 永久性访问地址/域名
  3. 利用大量服务实现可扩展性

客户机特点

  1. 与服务器通信，使用提供的服务
  2. 间歇性接入网络
  3. 可能使用动态ip地址
  4. 不会与其他客户机直接通信

P2P结构

  1. 没有永远在线的服务器
  2. 任意端系统/节点之间可以直接通讯
  3. 节点间歇性接入网络
  4. 可能改变ip地址

网络应用进程通信

同一主机上的进程通过操作系统提供通信机制，不同主机上运行的进程使用消息交换，客户机进程是发起通信的进程，服务器进程是等待通信请求的进程。

Socket
应用层到传输层之间的接口。进程通过Socket抽象使用下层协议提供的服务。

进程寻址
使用ip + 端口号，ip唯一标识了主机，端口号唯一标识进程。

应用层协议

公开协议
由RFC定义，例如http，遵循相同协议的不同网络应用可以互操作。=

私有协议
多数P2P文件使用私有协议

内容

  • 类型：请求消息、响应消息
  • 语法：消息中有哪些字段，如何描述
  • 语义：信息的含义
  • 规则：何时/如何发送并相应消息

Web 与 HTTP

特征：网页，网页互相链接

网页

  • 对象：HTML文件、JPEG图片、视频文件、动态脚本等
  • 基本HTML文件：包含对其他对象引用的链接

对象的寻址：

  • URL(Uniform Resource Locator)：统一资源定位器
  • 基本格式：Scheme://host:port/path（协议，主机号，端口号，路径）

协议

超文本传输协议：HyperText Transfer Protocol

C/S结构：

  • 客户-Browser：请求、接收、展示Web对象
  • 服务器-Web Server：响应客户的请求，发送对象

传输层协议：TCP

  • 服务器在80端口等待客户请求
  • 浏览器发起到服务器的TCP连接（创建套接字Socket）
  • 服务器接受来自浏览器的TCP连接
  • 浏览器与服务器交换HTTP消息
  • 关闭TCP连接

特点：无状态，服务器不维护任何有关客户端过去所发请求的信息。

连接类型

  1. 非持久性连接
    • 每个TCP连接最多允许传输一个对象
    • HTTP 1.0 使用
  2. 持久性连接
    • 每个TCP连接允许传输多个对象，发送响应后，服务器保持TCP连接的打开
    • HTTP 1.1 使用

连接过程：
非持久性连接：

  1. 发起、建立TCP连接：1个RTT
  2. 发送HTTP请求消息到响应消息的前几个字节到达：1个RTT
  3. 响应消息中所含的文件/对象传输时间

无流水的持久性连接：

  1. 客户端只有收到前一个响应才发送新的请求
  2. 每个对象耗时1个RTT

带有流水机制的持久性连接

  1. 客户端只要遇到一个引用对象就尽快发出请求
  2. 同时请求的所有对象只耗费1个RTT

消息格式

两类消息：请求消息（request）、响应消息（response）。使用ASCII码，人直接可读。

请求消息：
第一行：request line: 请求命令+url+http版本

前几行：header lines: 键值对，包括Host,User-agent(浏览器类型),Connection,Accept-language

空行，表示头部结束。
最后是请求的数据（Entity Body）。

请求的数据可以使用POST或GET上传。

  • POST方法：例如表格中，将消息体放在Entity Body中上传
  • GET方法：输入信息通过URL字段上传

方法的类型：GET、POST、HEAD、PUT、DELETE。其中HEAD请求只返回消息，不返回请求对象。PUT将消息体中的文件上传到URL字段指定的路径。DELETE删除URL字段指定的文件。

响应消息：
第一行：status line：协议+状态码+状态语句
前几行：header lines: 键值对，包括Connection，Date，Server，Last-Modified，Content-Length，Content-Type。
空行区分
最后是请求的文件。

Cookie

某些网站为了辨别用户身份，进行session跟踪而存储在用户本地终端上的数据（通常经过加密）。

Cookie的组件：

  • HTTP响应消息的cookie头部行
  • HTTP请求消息的cookie头部行
  • 保存在客户端主机上的cookie文件，由浏览器管理
  • Web服务器端的后台数据

Web缓存/代理服务器技术

功能：在不访问服务器的前提下，满足客户端的HTTP请求

  • 缩短客户请求的响应时间
  • 减少机构/组织的流量
  • 在大范围内（Internet）实现有效的内容分发

用户设定浏览器通过缓存进行Web访问。浏览器向缓存/代理服务器发送所有的HTTP请求

  • 如果所请求对象在缓存中，缓存返回对象
  • 否则，缓存服务器向原始服务器发送HTTP请求，获取对象，然后返回给客户端并保存该对象

缓存既充当客户端，也充当服务器，一般由ISP架设。

条件性GET方法
在HTTP请求消息中声明所持有版本的日期：If-modified-since:<data>，如果服务器中缓存的版本是最新的，则响应消息中不包含对象，返回条件码304: Not Modified。

Email应用

构成：

  • 邮件客户端：
    • 读写Email消息
    • 与服务器交互，收、发Email消息
  • 邮件服务器
    • 邮箱：存储发给该用户的Email
    • 消息队列：存储等待发送的Email
  • SMTP协议（Simple Mail Transfer Protocol）
    • 邮件服务器之间传递消息所使用的协议
    • 客户端：发送消息的服务器
    • 服务端：接收消息的服务器

SMTP

  • 使用TCP
  • 端口25
  • 不使用中间邮件服务器发送邮件
  • 传输过程的三个阶段
    • 握手
    • 消息的传输
    • 关闭
  • 命令/响应交互模式
    • 命令：ASCII文本
    • 响应：状态代码和语句
  • Email消息只能包含7位ASCII码
  • 使用持久性连接
  • 要求消息必须由7位ASCII码构成
  • SMTP服务器利用CRLF确定消息的结束。

文本消息格式：

  • 头部行：
    • To
    • From
    • Subject
  • 消息体
    • 消息本身（只能是ASCII字符）

MIME：多媒体邮件扩展

  • 通过在邮件头部增加额外的行以声明MIME的内容类型
  • 将非ASCII码数据编码后传输，接收方解码后还原。

邮件访问协议

通过邮件访问协议从服务器获取邮件。

  • POP: Post Office Protocol [RFC 1939]
    • 认证/授权（客户端<–>服务器）和下载
  • IMAP: Internet Mail Access Protocol [RFC 1730]
    • 更多功能，更加负载，能够操纵服务器上存储的信息
  • HTTP: 163, QQ Mail等。

POP3协议

  • 端口110
  • 认证过程（特许阶段）
    • 客户端命令：
      • User: 声明用户名
      • Pass: 声明密码
    • 服务器响应
      • +OK
      • -ERR
  • 事务阶段
    • List: 列出消息数量
    • Retr: 用编号获取消息
    • Dele: 删除消息
    • Quit

IMAP

  • 所有消息统一保存在服务器
  • 允许用户利用文件夹组织消息
  • 未发出删除命令前，一直保存在邮件服务器
  • 支持跨会话（Session）的用户状态：
    • 文件夹的名字
    • 文件夹与消息ID之间的映射

DNS应用

DNS(Domain Name System)，域名解析系统，解决了互联网上主机/路由器的识别问题：映射域名与IP地址。能够将域名翻译为IP地址。

  • 多层命名服务器构成的分布式数据库
  • 应用层协议：完成名字的解析
  • 负载均衡：Web服务器
  • 使用UDP协议
  • 端口53
  • 通常由其他协议间接使用

客户端想要查询www.someschool.edu的IP：

  • 同一台主机上运行着DNS应用的客户端
  • 浏览器提取主机名，传给DNS应用的客户端
  • DNS客户向服务器发送一个请求
  • DNS客户收到一个回答报文
  • 浏览器收到来自DNS的该IP地址

域名服务器

单个服务器有如下问题

  • 单点故障时，影响全网
  • 通信容量过大
  • 远距离集中式数据库：时延增加
  • 维护：单个服务器为所有主机保存记录

所以使用分布式、层次式数据库

过程：

  • 访问根服务器
  • 访问顶级域名服务器
  • 访问权威域名服务器
  • 交由本地缓存

DNS根域名服务器
本地域名服务器无法解析域名时，访问根域名服务器

  • 如果不知道映射，访问权威域名服务器
  • 获得映射
  • 向本地域名服务器返回映射

TLD和权威域名解析服务器
负责com,org,net,edu等顶级域名和国家顶级域名，例如cn,uk,fr。

权威域名服务器
组织的域名解析服务器，提供组织内部服务器的解析服务。

本地域名解析服务器

  • 不严格属于层级体系
  • 每个ISP有一个本地域名服务器（默认域名解析服务器）
  • 当主机进行DNS查询时，查询被发送道本地域名服务器，作为代理，将查询转发给（层级式）域名服务器

DNS记录缓存和更新

只要域名解析服务器获得域名-IP映射，即缓存这一映射

  • 一段时间过后，缓存条目失效（删除）
  • 本地域名服务器一般会缓存顶级域名服务器的映射，因此根域名服务器不经常访问。

DNS记录和消息格式

资源记录（RR,resource records）
RR format: (name, value, type, ttl)

  • Type=A
    • Name: 主机域名
    • Value: IP地址
  • Type=NS
    • Name: 域（edu.cn）
    • Value: 该域权威域名解析服务器的主机域名
  • Type=CNAME
    • Name: 某一真实域名的别名
    • Value: 真实域名
  • Type=MX
    • Value是与name相对应的邮件服务器

DNS协议：查询和回复消息，消息格式相同

  • 消息头部：
    • Identification: 16位查询编号，回复使用相同的编号
    • flags: 查询或回复、期望递归、递归可用、权威回答
    • 问题数、各类RR数量
  • 内容：
    • questions:Name,Type
    • answers: RR
    • authority: records for authoritative servers

P2P 应用

文件分发

假设一个模型：服务器提供一个大小为F的文件分发给N个客户端，上传时间为us，单个客户端下载时间为di。

在C/S架构中，总时间为max{NF/us,F/min{di}}，与客户端数量线性相关。

在P2P架构中，服务器必须发送一个副本，时间 F/us，客户机需要 F/di 时间下载，总共需要下载 NF 比特。所有主机一起上传，最快的可能上传速率为 us + ∑ui，总时间为 max{F/us,F/min{di},NF/(us+∑ui)}。

DHT

索引技术

  • P2P系统的索引：信息到接地那位置（IP地址+端口号）的映射。
  • 文件共享：
    • 利用索引动态跟踪节点所共享的文件的位置
    • 节点需要告诉索引它拥有哪些文件
    • 节点搜索索引，从而能够得到哪些文件
  • 即时消息
    • 索引负责将用户名映射到位置
    • 当用户开启IM应用时，需要通知索引它的位置
    • 节点检索索引，确定用户的IP地址

集中式索引

  1. 节点加入时，通知中央服务器：IP地址、内容
  2. 通过中央服务器查找
  3. 通过P2P获取内容

内容和文件传输是分布式的，但是内容定位是高度集中式的。

洪泛式查询（Query flooding）

  • 完全分布式架构
  • 每个节点对它共享的文件进行索引，且只对它共享的文件索引

覆盖网络：节点XY之间如果有TCP连接，那么构成一个边。所有活动节点和边构成覆盖网络，节点一般邻居数小于十个。

查询通过已有的TCP连接发送，节点转发查询消息。如果查询命中，则利用反向路径返回查询节点。

层次式覆盖网络

  • 介于集中式索引和洪泛查询之间的方法
  • 每一个节点或者是超级节点，或者被分配一个超级节点
    • 节点和超级节点间维持TCP
    • 某些超级节点之间维持TCP
  • 超级节点负责跟踪子节点的内容

Socket编程

应用编程接口

  • 数据链路层
    • 直接网卡编程
    • 基于Packet Driver编程
  • 网络层
    • 基于NDIS网络编程（Winodws）
    • 基于LibPcap/WinPcap、Libnet、Libnids、Libicmp编程
  • 传输层
    • NetBIOS编程（Windows）
    • Socket编程
  • 应用层
    • Web/RPC/中间件编程

Berkeley UNIX操作系统定义了Socket(套接字)，微软形成了一个稍有不同的API，称之为WINSOCK。

Socket是网络应用最典型的API接口，使用客户/服务器模型。使用IP地址+端口号唯一标识一个套接字。操作系统对内使用套接字描述符（socket descripter）管理套接字。套接字的抽象类似于文件。

地址结构
使用TCP/IP协议簇的网络应用程序声明端点地址变量时，使用sockaddr_in

WinSock

int WSAStartup(WORD wVersionRequested, LPWSADATA IpWSAData)。使用Socket的程序在使用之前必须首先调用WSAStartup函数。第一个参数指明程序请求使用的WinSock版本，其中高位字节指明副版本，低位字节指明主版本，例如0x102表示2.1版（十六进制）。第二个参数返回实际的WinSock版本信息，指向WSADATA结构的指针。

int WSACleanup(void);解除与Socket库的绑定，释放Socket库所占用的系统资源。

int socket(protofamily, type, proto)函数创建套接字，返回套接字描述符。

第一个参数是协议族，PF_INET表示TCP/IP，第二个参数表示套接字类型，type=SOCK_STREAM, SOCK_DGRAM or SOCK_RAW，第三个参数表示协议号，默认0。

套接字类型中SOCK_STREAM（流式）面向TCP，SOCK_DGRAM（数据报）面向UDP，SOCK_RAW面向网络层协议。

  • TCP：可靠、面向连接、字节流传输、点对点
  • UDP：不可靠、无连接、数据报传输

int closesocket(SOCKET sd);关闭一个描述符为sd的套接字。如果多个进程共享一个套接字，调用其将套接字引用计数减1，减至0才关闭。一个进程中的多线程对其不计数。

int bind(sd,localaddr,addrlen)，绑定套接字的本地端点地址（IP+端口号）。参数：

  • 套接字描述符
  • 本地端点地址（socketaddr_in）

客户端程序一般不必调用bind函数。服务器段需要调用设置端口号。服务器的IP地址可以使用地址通配符INADDR_ANY表示主机的任意一个IP地址。

int listen(sd,queuesize)服务器套接字置于被动监听状态，仅用于服务器端，面向连接的流套接字。queuesize表示缓存队列的大小。

connect(sd,saddr,saddrlen)只用于客户端，调用connect来连接客户套接字sd和特定端口saddr。

accept(sd,caddr,caddrlen)只用于服务器端TCP套接字，从listen缓存队列中取出请求，创建新套接字与客户通信。

send(sd,*buf,len,flags) sendto(sd,*buf,len,flags,destaddr,addrlen)发送数据。

recv(sd,*buffer,len,flags) recvfrom(sd,*buf,len,flags,senderaddr,saddrlen)接收数据。

setsockopt getsockapt设置socket参数。

网络字节顺序

TCP/IP定义了标准的独立于操作系统的网络字节顺序。某些Socket函数的参数需要存储为网络字节顺序（如IP地址、端口号等）。

  • htons：本地字节顺序->网络字节顺序（16bits）
  • ntohs：网络字节顺序->本地字节顺序（16bits）
  • htonl：本地字节顺序->网络字节顺序（32bits）
  • ntohl：网络字节顺序->本地字节顺序（32bits）

Socket基本流程

服务器端：

  1. （WINDOWS下）调用WSAStartup
  2. s = socket()
  3. bind(s)
  4. listen(s)
  5. ns = accept(s)（阻塞）
  6. recv(ns) || send(ns)
  7. closesocket(ns)
  8. back to 5
  9. （WINDOWS下）WSACleanup

客户端：

  1. （WINDOWS下）调用WSAStartup
  2. s = socket()
  3. connect(s)（阻塞）
  4. send(s) || recv(ns)
  5. closesocket(s)
  6. （WINDOWS下）WSACleanup

解析

客户端可能使用域名或IP地址标识服务器，IP协议需要32位二进制IP地址，函数inet_addr()实现点分十进制IP地址到32位IP地址转换，函数gethostbyname()实现域名到32位IP地址转换，返回一个指向结构hostent的指针。

客户端还可能使用服务名标识服务器端口。需要将服务名转换为熟知端口号，使用函数getservbyname()。

所有的协议都使用协议号进行标识。客户端可能使用协议名指定协议，例如TCP->6。函数getprotobyname()实现协议名到协议号的转换。

TCP客户端软件流程

  1. 确定服务器IP地址与端口号
  2. 创建套接字
  3. 分配本地端点地址（IP地址+端口号，操作系统自动完成）
  4. 连接服务器（套接字）
  5. 遵循应用层协议进行通信
  6. 关闭/释放连接

UDP客户端软件流程

  1. 确定服务器IP地址与端口号
  2. 创建套接字
  3. 分配本地端点地址
  4. 指定服务器端点地址，构造UDP数据报
  5. 遵循应用层协议进行通信
  6. 关闭/释放套接字

服务器软件设计

四种基本类型服务器：

  • 循环无连接
    1. 创建套接字
    2. 绑定端点地址（INADDR_ANY+端口号）
    3. 反复接收来自客户端的请求
    4. 遵循应用层协议，构造响应报文，发送给客户
  • 循环面向连接服务器
    1. 创建主套接字，并绑定熟知端口号
    2. 设置（主）套接字为被动监听模式，准备用于服务器
    3. 调用accept()函数接收下一个连接请求，通过新套接字用于与该客户建立连接
    4. 遵循应用层协议，反复接收客户请求，构造并发送响应
    5. 完成为特定客户服务后，关闭与该客户之间的连接，返回步骤3
  • 并发无连接服务器基本流程
    主线程
    1. 创建套接字，并绑定熟知端口号
    2. 反复接收下一个客户请求，并创建新线程处理该客户响应
    子线程
    1. 接收一个待定请求
    2. 依据应用层协议构造响应报文并发送
    3. 退出
  • 并发面向连接服务器基本流程
    主线程
    1. 创建（主）套接字，并绑定熟知端口号
    2. 设置（主）套接字为被动监听模式，准备用于服务器
    3. 反复接收下一个请求，并创建一个新的子线程处理该客户响应
    子线程
    1. 接收一个客户的服务请求
    2. 遵循应用层协议与特定客户交互
    3. 关闭/释放连接并退出
